04 D4: Flow — 业务流程编排（核心骨架）★
★ 核心骨架设计哲学

Agent 是智能但不可预测的执行单元。Flow 是让 Agent 的行为变得可预测、可控制、可重复的编排骨架。没有 Flow，Agent 只能做"自由对话"；有了 Flow，Agent 才能稳定完成真实业务。

4.1核心概念
FlowTemplate (业务流程定义) FlowInstance (运行时实例) ├── steps[]: 有序步骤定义 ├── status: 状态机 │ ├── node_id ├── execution_state: 各步骤运行时状态 │ ├── node_type: (见下表) ├── context_bag: 步骤间数据传递 │ ├── agent_id / agent_selector ├── current_step / total_steps / progress │ ├── acceptance_criteria ├── pending_action: HITL 暂停点 │ ├── completion_mode └── origin: 触发来源 │ ├── context_output_key │ ├── hitl_policy: (该步骤的 HITL 策略) │ └── edges[]: DAG 拓扑边 ├── trigger_config: 触发方式 ├── execution_mode: template | dynamic └── template_status: draft → published → archived

4.4三大核心能力
能力一：结构化步骤编排
FlowTemplate 定义了"做什么业务"和"按什么顺序做"： 用户意图: "帮我做一份市场调研报告" │ ▼ Flow 自动编排: ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Step 1 │──→│ Step 2 │──→│ Step 3 │──→│ Step 4 │ │ 信息搜集 │ │ 数据分析 │ │ HITL审批 │ │ 报告生成 │ │ Agent(A) │ │ Agent(B) │ │ 人工审批 │ │ Agent(A) │ │ AC: 5条 │ │ AC: 3条 │ │ │ │ AC: 格式 │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ │ ▼ ▼ ▼ context_bag["research_data"] → context_bag["analysis_result"] → 最终输出

关键设计：步骤-任务解耦 — Step 是编排单位（Flow 模板中定义），Task 是可选的执行工件（运行时按需创建）。不是所有步骤都需要 Task——HITL 节点、条件分支、外部等待都不产生 Task。

能力二：步骤间数据传递 — Context Bag
Context Bag 是 Flow 的" bloodstream "——在步骤之间传递结构化业务数据： Step 1 (研究) ──写入──→ context_bag["research_output"] = {...} │ Step 2 (分析) ←──读取─────┘ (只读，不可修改前置步骤的输出) │ ──写入──→ context_bag["analysis_result"] = {...} │ Step 3 (审批) ←──读取─────┘ 人工审批时可以看到前置步骤的所有输出 │ Step 4 (生成) ←──读取─────┘ 汇总所有前置上下文生成最终报告 规则： - 每个 key 只能写入一次（write-once 语义） - 值经过净化（防止 Prompt Injection） - 单步骤上限 50KB，总计上限 1MB
能力三：验收标准（AC）驱动的质量保障
每个步骤可以定义 Acceptance Criteria (AC): AC 来源 (三层合并，优先级从高到低): 1. Template AC: FlowTemplate 中步骤级定义的验收标准 2. Boss AC: 运行时由发起人/管理员覆盖的标准 3. Agent AC: Agent 自身声明的能力标准 AC 验证: ├── 里程碑检查: 输出是否包含关键交付物 ├── 格式检查: 输出是否符合指定格式 ├── 质量检查: 输出是否满足质量阈值 └── 自定义检查: 可扩展的验证器 AC 失败处理: ├── 重试: 重新执行该步骤 (max_retries) ├── 跳过: 标记为 "skipped"，继续推进 ├── 升级: 转为 HITL 节点，交由人工处理 └── 失败: 终止 Flow

4.7为什么 Flow 是 Agent 稳定完成业务的关键

没有 Flow

  • 执行路径：Agent 自由发挥
  • 质量保障：无
  • 人工介入：只能对话中请求
  • 数据传递：仅靠记忆
  • 可重复性：每次结果不同
  • 可观测性：只有对话日志
  • 错误处理：靠 Agent 自愈
  • 业务适配：靠 Prompt 工程
→

有 Flow

  • 结构化步骤编排
  • AC 验证 + 重试
  • 原生 HITL 节点（审批/协同）
  • Context Bag（结构化）
  • 相同输入 → 相似结果
  • 步骤级状态 + 进度 + 审计
  • AC 失败 → 重试/升级/人工
  • Flow 模板 + AC + HITL 策略

结论：Agent 负责智能执行，Flow 负责业务保障。两者结合 = 灵活性 + 可靠性

05 D5: Task + HITL — 执行工件与治理检查点
D5核心定位 — Flow 编排体系中的两个侧面
Flow (业务流程) │ ├── 每个 Step 可能产生一个 Task (执行工件) │ Task 记录: 做了什么、进度如何、结果是什么 │ └── 每个 Step 可能有 HITL 检查点 (治理门控) HITL 记录: 谁审批了、何时审批、审批结果
Task 生命周期
CREATED ──(规划)──→ PLANNED ──(启动)──→ EXECUTING ──(提交)──→ REVIEW │ │ │ │ │ │ ↓ ↓ └──(取消)──→ CANCELLED PAUSED ──(恢复)──→ EXECUTING │ ↓ AWAITING_APPROVAL ┌──┴──┐ (批准) (拒绝) │ │ ↓ ↓ EXECUTING EXECUTING (带反馈)
  • Flow 驱动：由 Flow 步骤创建（flow_instance_id + step_order）
  • Plan 驱动：由 PLAN_EXECUTE 分解产生（parent_task_id 父子关系）
  • 独立创建：用户/Agent 直接创建，不依赖 Flow
D5HITL — 统一 ApprovalGate，嵌入 Flow 治理
ApprovalGate（统一审批门控）
ApprovalGate ├── trigger: │ ├── TOOL_CALL 工具调用风险过高 │ ├── STEP_CHECKPOINT Flow 步骤执行到检查点 │ ├── AC_FAILURE AC 验证失败，需要人工裁决 │ ├── PLAN_REVIEW PLAN_EXECUTE 计划需审批 │ └── SENSITIVE_ACTION 涉及敏感数据/操作 │ ├── scope: TOOL_LEVEL | STEP_LEVEL | TASK_LEVEL │ ├── policy (由 Flow Step 的 hitl_policy 决定): │ ├── AUTO_APPROVE 自动批准（仅记录审计） │ ├── NOTIFY_THEN_PROCEED 通知后继续（异步通知） │ ├── REQUIRE_APPROVAL 必须等待批准 │ └── REQUIRE_REASON 必须填写理由后批准 │ ├── state: PENDING → APPROVED | REJECTED | EXPIRED | CANCELLED ├── approver: {user_id, role_requirement} ├── context: 执行状态快照（用于恢复） └── timeout: {duration, escalation_policy}
HITL 在 Flow 中的三种形态
形态 1: 工具级 HITL (自动触发) ┌──────────┐ │ Step 2 │ Agent 执行中调用高风险工具 │ Agent(B) │──→ 工具触发 ApprovalGate (trigger=TOOL_CALL) │ │ Flow 暂停到 awaiting_approval │ │ 用户批准后恢复执行 └──────────┘ 形态 2: 步骤级 HITL (流程设计时定义) ┌──────────┐ │ Step 3 │ FlowTemplate 中定义为 hitl_approval 节点 │ 人工审批 │ 必须人工决策才能流转 │ │ 用户看到前置步骤的 context_bag 输出 │ │ 批准/拒绝 → Flow 继续推进 └──────────┘ 形态 3: 协同级 HITL (人机共创) ┌──────────┐ │ Step 3 │ FlowTemplate 中定义为 hitl_cocreation 节点 │ 人机协同 │ 人工主导，Agent 辅助 │ │ 多轮交互直到双方满意 │ │ 完成后 → Flow 继续推进 └──────────┘
06 D3: 运行时引擎
D3统一引擎 + 三种执行模式
统一 RuntimeEngine ├── REACT 模式 (单步推理循环) ├── PLAN_EXECUTE 模式 (规划-执行管道) └── HYBRID 模式 (复杂度自适应路由)
模式流程适用场景
REACT观察 → 组装上下文 → LLM推理 → 工具调用? → [是] → 执行工具 → 回到推理 / [否] → 输出响应单步推理循环
PLAN_EXECUTE规划: 复杂任务 → 分解为子任务树/子Flow → 分配 Agent → [可选] 用户审批计划
执行: 按序/并行执行 → 每个子任务用 REACT → 检查点触发 HITL
回顾: 汇总结果 → 呈交审阅 → 收集反馈迭代		规划-执行管道
HYBRID复杂度评估 → SIMPLE 路由到 REACT / COMPLEX 路由到 PLAN_EXECUTE
模糊时默认 REACT，后台影子模式对比优化		复杂度自适应路由
统一执行契约：所有模式产出相同的事件流
TEXT_DELTA → LLM 文本输出 TOOL_CALL → 工具调用请求 TOOL_RESULT → 工具执行结果 STATE_WRITE → 状态变更（任务更新、记忆写入） CHECKPOINT → 计划进度检查点 HITL_REQUEST → 需要人工审批 ERROR → 执行错误 COMPLETE → 执行完成
07 D6: 模型接入层
D6每轮动态路由
Agent.model_preferences → 每轮 ModelRouter 决策 → 可中途切换
ModelRouter 决策流程
输入: agent偏好 + 任务复杂度 + 上下文Token需求 + 成本预算 + 能力需求 + 用户权限 ├── 1. 能力过滤: 排除不满足能力需求的模型 ├── 2. 窗口过滤: 排除上下文窗口不足的模型（来自 D2 预算） ├── 3. 权限过滤: 排除无权使用的模型（来自 D8） ├── 4. 成本-性能优化: 简单→高效模型，复杂→强模型 ├── 5. 负载均衡: 多候选选最低负载 ↓ 输出: {selected_model, routing_reason, fallback_chain}

Flow 与模型路由的协作：Flow 的不同步骤可以路由到不同模型 — 规划步骤用强模型，数据处理步骤用高效模型，审批节点不需要模型。

08 D7: 外部服务集成层
D7统一集成网关
IntegrationGateway（统一入口） ├── IntegrationRegistry (工具注册中心) │ ├── MCP Server 连接时自动发现注册 │ ├── OpenAPI/Swagger 导入自动生成 │ ├── REST API 动态注册 │ └── 工具版本化 ├── IntegrationRouter (tool_name → 确定来源适配器) ├── IntegrationExecutor (统一执行: Builtin/DB/Connector/MCP/Webhook) └── IntegrationGovernance (限流/审计/凭据/健康监控/熔断)

09 D8: 统一权限设计
D8四层权限管线 + Flow 级权限
Layer 1: 用户权限 — 谁能做什么 Action: create | read | update | delete | execute | approve Resource: agent | task | flow | tool | model | connector | document | memory Scope: GLOBAL | ORG | OWN_ONLY Layer 2: Agent 权限 — Agent 执行时能做什么 tool_access + data_access + action_limit Layer 3: Agent + 服务权限 — Agent 通过特定服务能做什么 allowed_operations + data_scope + rate_limit Layer 4: 服务自身权限 — 服务需要什么、合规要求 auth_requirements + data_requirements + compliance
Flow 级权限
FlowPermission ├── 谁可以启动此 FlowTemplate (execute 权限) ├── 谁可以审批此 Flow 中的 HITL 节点 (approve 权限) ├── 此 Flow 可以访问哪些数据范围 (data_scope) └── 此 Flow 的步骤可以调用哪些外部服务 (service_bindings)
权限评估管线
用户启动 Flow ├── L1: 用户能否执行此 FlowTemplate? ├── Flow 执行中，Step 2 的 Agent 调用 Connector Z 的 Operation Y: │ ├── L2: Agent 能否使用此工具? │ ├── L3: Agent 能否通过 Connector Z 调 Operation Y? │ └── L4: Connector Z 的认证和合规是否满足? └── Flow Step 3 是 HITL 审批节点: └── L1: 审批人角色是否满足 approve 权限要求?
