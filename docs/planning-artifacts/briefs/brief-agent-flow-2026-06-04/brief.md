# Product Brief: Agent Flow

## Executive Summary

Agent Flow 是一个面向 MES（制造执行系统）场景的 AI Agent 编排平台。它让工厂的 IT 团队能快速搭建 AI 工作流，让产线工程师能不写代码地配置和使用 AI Agent——把 AI 能力变成 MES 日常运营的一部分。

今天的通用 AI 平台（Dify、LangFlow）都不理解 MES：没有工业协议支持、没有 MES 数据模型、没有产线工程师能用的界面。Agent Flow 不做通用 AI 平台的另一个竞品，而是做 MES 领域原生的 Agent 编排工具——内置工业数据接入、MES 业务组件和分角色体验。

内部工具定位，几十人规模，支持工厂内网和公司私有云部署。

## 产品背景

MES 系统产线上每天都在发生需要人工判断和处理的事件——质量异常、设备报警、工艺参数偏移、工单状态变更。这些场景天然适合 AI 辅助，但现状是：

- **IT 团队做 AI 集成成本高**：用通用平台（Dify/LangFlow）做 MES 场景，需要从零构建工业数据接入、协议解析、业务映射，大量定制开发
- **产线工程师被排除在外**：现有工具全是技术界面（LLM Node、RAG Pipeline、Embeddings），产线工程师看不懂也用不了，只能依赖 IT 团队
- **AI 和 MES 是两个世界**：通用 AI 平台不理解工单、设备、工艺路线、质量批次这些 MES 概念，AI 和业务之间隔着一层翻译

**为什么是现在**：AI Agent 技术已经成熟到可以实际落地工业场景，但通用平台和 MES 之间的鸿沟还没有人填。服务 MES 的团队拥有领域知识优势，这是做领域原生工具的最佳时机。

## 产品功能

### 1. Agent 配置与管理

**是什么**：Agent 是无状态的角色配置，由提示词模板、工具组合、知识库和运行时参数组成。Agent 之间不直接通信，只通过 Workflow 协作。

**为什么这么做**：将 Agent 设计为无状态配置而非有状态实体，是为了让同一个 Agent 角色可以被多个 Workflow 和对话复用，不绑定到特定的会话上下文。这也意味着 Agent 的创建、修改、复制都很轻量——改一下配置就是一个新 Agent。

**意义**：IT 团队可以快速定义不同角色的 Agent（质量分析员、设备诊断专家、工单调度助手），产线工程师可以在操作员视图中直接选用。

### 2. Workflow 编排

**是什么**：可视化的工作流编排引擎，底层统一为有环状态图。前端提供两种编辑模式——简单模式（DAG，无环有向图）面向非技术用户，高级模式（状态图，支持循环和回退）面向开发者。

**为什么这么做**：MES 场景中有些流程是线性的（报警→分析→通知），DAG 就够了；有些流程需要循环和回退（如多轮质量检测、设备故障排查的迭代诊断），需要状态图。两种模式共享同一套底层执行引擎，前端按角色展示不同复杂度。

**意义**：产线工程师用简单模式拖拽连线就能搭建线性流程，IT 团队用高级模式处理复杂逻辑。不是两套系统，是一套系统的两个界面。

### 3. 工具系统

**是什么**：工具与 Agent 解耦，独立存在，Agent 自由选择组合。工具来源有三种：
- **MCP**：外部协议连接，配置即用，对接已有的系统和服务
- **Skill**：Markdown 格式的技能描述，支持 Git 拉取、目录上传、前端创建，归一化为同一格式
- **子工作流**：Workflow 自动注册为工具，实现流程复用

**为什么这么做**：工具与 Agent 解耦是系统扩展性的基石。通用 AI 平台的 Agent 和工具是绑定的，换工具就要重新配 Agent。我们让工具独立注册、Agent 自由组合，同一个"设备状态查询"工具可以被质量分析 Agent 和设备维护 Agent 共同使用。Skill 的 Markdown 归一化降低了工具的创建门槛——写个 md 文件就是一个工具。

**意义**：工具生态可以独立增长，不依赖 Agent 的变化。MES 场景中常见的工具（设备数据查询、工单操作、质量数据检索）可以逐步积累形成领域工具库。

### 4. MES 领域组件

**是什么**：内置 MES 业务概念作为一等公民——工单、设备、工艺路线、质量批次、报警等。提供 MES 领域的预置组件（设备状态查询、质量数据检索、工单创建、报警升级）和行业模板（质量异常分析 Agent、设备预测维护 Agent、工艺参数优化 Agent）。

**为什么这么做**：通用平台的组件库面向 AI 场景（LLM、RAG、向量搜索），产线工程师看到 "ChatInput"、"OpenAIEmbeddings" 完全不知道什么意思。我们用 MES 术语命名组件——"设备监控"、"质量检查"、"报警通知"——让领域用户直接理解。

**意义**：这是与 Dify/LangFlow 的核心差异。不是"在通用 AI 平台上做 MES"，而是"为 MES 场景量身定做的 Agent 平台"。用户不需要把 MES 业务逻辑翻译成 AI 概念，平台本身就理解 MES。

### 5. 工业数据接入

**是什么**：原生支持工业协议（OPC-UA、MQTT、Modbus）和 MES 系统数据接口，Agent 和 Workflow 可以直接消费结构化工业数据（时序数据、设备状态、工艺参数），而不是只能处理文档。

**为什么这么做**：Dify 的 RAG 系统以文档检索为核心设计，LangFlow 没有原生的工业连接器。MES 场景中最重要的数据不是 PDF 文档，而是实时的设备状态和工艺参数。如果 AI 只能读文档而不能感知产线实时状态，那它就只是个知识库，不是运营工具。

**意义**：Agent 能实时监听设备数据变化并主动响应——设备温度超标自动触发诊断流程、质量数据异常自动升级报警。从"人问 AI 答"变成"AI 主动发现和处理"。

### 6. 分角色用户体验

**是什么**：同一平台根据用户角色展示不同界面：
- **开发者视图**：类似 LangFlow 的自由拖拽编辑器，完整的节点配置、条件边、数据端口
- **操作员视图**：表单/向导式界面，用 MES 术语描述操作，隐藏技术细节

**为什么这么做**：MES 场景的用户跨度大——IT 团队熟悉技术概念，产线工程师只懂业务。通用平台只有一个技术界面，导致产线工程师被完全排除在 AI 工具之外。角色化界面让不同用户看到适合自己的操作方式，而不是一个界面所有人迁就。

**意义**：产线工程师能自主配置和使用 AI Agent，不再事事依赖 IT 团队。这才是 AI 真正融入 MES 日常运营的关键——如果只有 IT 能用，那 AI 永远只是 IT 项目，不是业务工具。

### 7. 对话与 Workflow 双向触发

**是什么**：用户对话可以触发已配置的 Workflow 执行，Workflow 中的 Agent 节点可以创建或继续对话。对话和 Workflow 各自生命周期独立。

**为什么这么做**：MES 场景中的 AI 交互不是纯对话也不是纯流程，而是混合的。工程师可能先和 Agent 对话描述问题，Agent 自动启动一个诊断 Workflow，Workflow 中的某个步骤又需要回来说"我需要你确认一下这个参数"。纯对话平台（Dify Chat）做不到流程编排，纯流程平台做不到灵活对话。

**意义**：覆盖 MES 场景中"人机协作"的真实模式——不完全是人在主导，也不完全是机器在跑，而是人和 AI 交替推进。

### 8. 执行引擎与可观测性

**是什么**：单机异步任务队列执行引擎，WebSocket 实时推送结果。支持完整执行日志、调用链追踪、断点续跑（从失败节点恢复）。双层循环保护（Workflow 级最大递归深度 + 全局调用链深度限制）。

**为什么这么做**：MES 系统对可追溯性有合规要求（质量追溯），AI 的每一次决策都必须有记录。断点续跑是因为工业场景中 Workflow 可能运行很长时间，不能因为一个节点失败就全部重来。循环保护是因为 Agent 和 Workflow 可以互相嵌套，没有保护就可能无限递归。

**意义**：Agent 的行为可审计、可追溯、可恢复——这是工业场景的基本要求，也是通用 AI 平台忽视的。

## 差异化定位

| 维度 | 通用平台 (Dify/LangFlow) | Agent Flow |
|------|------------------------|------------|
| 领域理解 | 无，需要从零构建 MES 映射 | MES 数据模型和业务组件内置 |
| 目标用户 | IT/开发者为主 | IT 团队 + 产线工程师，分角色界面 |
| 数据接入 | 文档和 API 为主 | 工业协议原生支持，结构化数据优先 |
| 部署 | 重（Dify 7+ 容器）或需外网 | 轻量，支持工厂内网离线部署 |
| 交互模式 | 对话式为主 | 事件驱动 + 对话式，双向触发 |
| 工具生态 | 通用 AI 工具 | MES 领域工具库 + MCP + Skill |

我们的壁垒不是某个技术突破，而是**领域专注**——不做通用 AI 平台的所有东西，只做 MES 场景真正需要的。

## 不做什么

为了保持聚焦，以下事情明确不在产品范围内：

- **通用 RAG 管线**：Dify 已经做得很好，我们不竞争
- **100+ LLM 模型支持**：MES 场景用 5-10 个模型足够
- **SaaS 多租户**：内部工具，不需要
- **通用 AI 应用市场**：初期用自定义组件 + MCP 扩展即可

## 谁在使用

**工厂 IT/开发团队**
搭建和维护 AI Agent 工作流，需要完整的开发者工具链（SDK、API、调试面板）。

**产线工程师**
使用和简单配置 AI Agent，不写代码，用 MES 领域语言操作。这是产品成功的决定性用户——如果他们能用，说明产品真正解决了问题。

**MES 系统管理员**
管理平台配置、用户权限、工具注册，关注系统稳定性和可追溯性。

## Vision

2-3 年后，Agent Flow 成为 MES 系统中 AI 能力的标准入口——就像今天 MES 系统有报表模块、看板模块一样，Agent 模块是标配。产线工程师每天打开 Agent Flow，像使用 Excel 一样自然地配置 AI 辅助流程。

从 MES 出发，这套领域原生的 Agent 编排能力可以延伸到其他工业场景——SCADA、ERP、PLM——成为工业 AI 应用的通用编排层。
