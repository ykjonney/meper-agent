# Decision Log: Agent Flow Product Brief

## 2026-06-04

### D001: 定位为 MES 领域原生而非通用 AI 平台
- **背景**: 市场研究显示 Dify 和 LangFlow 在 MES 场景有共同盲区
- **决策**: 不做通用平台竞争，做 MES 领域垂直
- **原因**: 通用平台已成熟，竞争价值低；MES 垂直场景有明确未被满足的需求

### D002: 分角色体验（开发者 + 操作员）
- **背景**: 目标用户包含 IT 团队和产线工程师两类
- **决策**: 借鉴 LangFlow 可视化体验，但做角色化适配
- **原因**: 产线工程师无法使用纯技术界面，但也不需要 Dify 那样的产品化界面

### D003: MVP 先做 DAG 简单模式
- **背景**: 头脑风暴确定了简单模式（DAG）+ 高级模式（状态图）的双模式设计
- **决策**: MVP 只做 DAG，状态图后续迭代
- **原因**: DAG 覆盖 80% 场景，状态图实现复杂度高，先验证核心价值

### D004: 壁垒是领域专注 + 执行速度
- **背景**: 没有独特的技术壁垒
- **决策**: 诚实承认，不虚构技术护城河
- **原因**: 内部工具定位，几十人规模，不需要技术壁垒，需要快速交付和领域适配

### D005: 轻量部署路线
- **背景**: 部署环境包括工厂内网和公司私有云
- **决策**: 走 LangFlow 的轻量路线，而非 Dify 的重量级架构
- **原因**: 工厂服务器资源有限，7+ 容器方案不现实

### D006: 产品简报聚焦"做什么、为什么做"，不划 MVP 范围
- **背景**: 初稿包含了 MVP 范围划分（包含/不包含）
- **决策**: 去掉 MVP 范围，改为逐个功能说明"是什么、为什么这么做、意义"
- **原因**: 产品简报是定义产品功能和价值的文档，MVP 划分是 PRD 的工作
