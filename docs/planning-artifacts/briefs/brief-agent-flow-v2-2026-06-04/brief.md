# Product Brief: Agent Flow

## Executive Summary

Agent Flow 是一个通用 AI Agent 编排平台。它让开发团队能在 Web 界面上创建和配置自主 Agent——为 Agent 组合工具、知识库、工作流等能力，然后通过任务驱动让 Agent 自主规划和执行。外部系统通过 API/SDK 接入，消费平台上的 Agent 能力。

核心设计哲学：**工作流是 Agent 的一种能力**。Agent 拿到任务后自主决策执行路径——可以走自由推理，也可以选择调用某个已配置的工作流走确定性流程。这让同一个 Agent 在不同场景下自动切换执行策略：需要灵活性时自主推理，需要确定性时走工作流。

平台不是又一个 Dify 或 LangFlow。差异在于 Agent 是真正的自主决策主体——它不是被工作流编排的执行节点，而是主动选择是否使用工作流的智能体。工作流是它的工具之一，不是它的牢笼。

## The Problem

今天构建 AI Agent 应用的团队面临三个核心痛点：

**Agent 能力单一且不可组合。** 现有平台上，一个 Agent 要么做对话，要么执行固定流程。想让它同时具备搜索、分析、审批、生成等多种能力，需要硬编码逻辑或复杂的手动串联。Agent 的能力是静态的——创建时就固定了，不能按需扩展。

**灵活性和确定性不可兼得。** 需要灵活推理的场景（研究、分析）和需要确定性执行的场景（审批流程、工业操作）在同一个系统中是割裂的。对话式 Agent 不可控，工作流 Agent 不灵活。没有统一的模型让一个 Agent 同时具备两种能力。

**Agent 能力无法被外部系统消费。** 团队搭好了一个 Agent，但要让它服务于公司的其他系统（MES、ERP、BI），还需要单独写集成代码。Agent 是孤立的，不是公司的能力中台。

## The Solution

**Agent 是可配置的能力集合。** 每个 Agent 由提示词、工具组合、知识库、工作流绑定、模型偏好等构成。开发人员在 Web 界面上配置 Agent 的能力，Agent 在运行时根据任务自主决定使用哪些能力。

**工作流是 Agent 的能力之一。** Agent 可以配置多个工作流。收到任务时，Agent 自主规划执行路径——如果判断任务适合某个已有工作流，就调用它走确定性流程；如果判断需要灵活处理，就走自由推理。两种模式在同一个 Agent 内无缝切换，由 Agent 自主决策。

**平台是 Agent 能力中台。** 所有 Agent 及其能力统一管理，外部系统通过 API/SDK 接入消费 Agent 能力。一个质量分析 Agent 可以被 MES 系统调用处理报警，也可以被 BI 系统调用生成报告——同一个 Agent，不同的消费方。

**完整 Web 管理界面。** 包含 Agent 创建与配置、工作流编排、Skill 和 MCP 管理、执行日志与可观测性、权限管理等全部功能。开发人员在 Web 界面完成所有操作，不需要写代码来管理 Agent。

## What Makes This Different

| 维度 | Dify / LangFlow | Agent Flow |
|------|----------------|------------|
| Agent 定位 | 被编排的执行节点 | 自主决策的智能体，工作流是它的能力 |
| 执行模式 | 对话 or 工作流，二选一 | Agent 自主选择推理 or 工作流，统一模型 |
| 能力组合 | Agent 创建时固定 | 按需配置，动态组合工具/知识库/工作流 |
| 集成方式 | 平台内闭环 | Agent 能力中台，API/SDK 对外输出 |
| 工作流角色 | 编排 Agent 的骨架 | Agent 可选的能力之一 |

核心壁垒不是某个技术突破，而是 **Agent 自主规划 + 工作流作为能力** 这个统一模型。Agent 不是被工作流驱动的木偶，也不是只能自由对话的聊天机器人——它是一个能自主判断"这个任务我该走流程还是自由发挥"的智能体。

## Who This Serves

**开发团队（平台建设者）**
在 Web 界面上创建和配置 Agent，为 Agent 组合工具、知识库、工作流等能力。管理 Agent 的生命周期和权限。需要完整的开发者工具链（API、SDK、调试面板、日志追踪）。

**业务系统（能力消费者）**
公司内部的 MES、ERP、BI 等系统通过 API/SDK 接入平台，调用 Agent 能力完成业务任务。不直接操作平台界面，而是通过接口消费 Agent 的推理、分析、生成等能力。

**平台管理员**
管理平台配置、用户权限、工具注册、模型接入。关注系统稳定性、可追溯性和资源管理。

## Success Criteria

- 开发人员能在 30 分钟内创建一个可用的 Agent 并配置基本能力
- Agent 能根据任务描述自主选择执行路径（推理 or 工作流），无需人工指定
- 外部系统通过 API 调用 Agent 能力，响应时间符合业务要求
- 工作流执行路径可追踪、可审计，满足工业场景的合规要求
- 多部门同时使用，Agent 能力隔离且可共享

## Scope

### 第一版（MVP）

- Agent 创建与配置（提示词、工具、知识库、工作流绑定）
- 工作流编排引擎（DAG 模式，步骤级编排）
- Agent 自主规划与执行（REACT 模式）
- 工具系统（MCP 接入、Skill 管理）
- 上下文管理与压缩
- 模型接入与动态路由
- API/SDK 接口（外部系统调用 Agent 能力）
- 基础权限管理
- 执行日志与可观测性
- Web 管理界面

### 明确不做

- SaaS 多租户和计费系统
- 应用市场 / 模板市场
- 移动端界面
- 100+ LLM 模型支持（5-10 个足够）
- 复杂的状态图模式（MVP 先做 DAG）

## Vision

2-3 年后，Agent Flow 成为公司内部的 AI 能力中台——各部门的业务系统通过 API 调用平台上的 Agent 能力，就像调用微服务一样自然。开发人员在 Web 界面上配置 Agent 和工作流，Agent 自主完成从信息搜集、分析决策到执行反馈的全流程。

从内部工具起步，这套"Agent 自主规划 + 工作流作为能力"的模型可以延伸为通用的 Agent 编排平台，服务更多组织的 AI 应用构建需求。
