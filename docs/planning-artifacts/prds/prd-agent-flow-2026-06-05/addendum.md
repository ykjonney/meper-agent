## A.1 架构层待定义项（来自头脑风暴）

以下设计决策由架构阶段负责定义，PRD 不做规定：

### 执行引擎架构
- 单机部署，异步任务队列（具体选型待定）
- WebSocket 实时推送执行进度和结果
- 任务级并发控制（限流策略）

### 类型系统
- 混合类型系统：内部 Schema + 变量引用 + 自动推断（参考 Dify 实现）
- 工作流节点输入/输出端口的类型推断和校验规则
- 类型不匹配时的错误处理策略

### 提示词变量来源体系
- 五种变量来源：Workflow 入参 / 上游节点输出 / 知识库检索结果 / 系统上下文 / 历史对话
- 变量解析优先级与冲突解决策略
- 统一的变量引用语法

### 条件表达式引擎
- 三层能力：节点输出判断 / 数据条件（变量表达式）/ 事件触发（后续迭代）
- 支持的运算类型和复杂度边界
- 自定义函数的支持范围

## A.2 市场竞品参考数据（来自市场调研）

### 部署架构基准
| 产品 | 架构 | 最小部署 |
|------|------|---------|
| Dify | Flask + Celery + Redis + PostgreSQL | 7+ Docker 容器 |
| LangFlow | FastAPI + SQLModel + React | 单进程可运行 |

Agent Flow 目标：借鉴 LangFlow 轻量路线，建议在架构阶段量化部署约束（如"最小部署依赖容器数 ≤ 3"）。

### 竞品差异化定位记录
| 维度 | Dify | LangFlow | Agent Flow 方向 |
|------|------|----------|----------------|
| Agent 定位 | 被编排的执行节点 | 组件流中的处理单元 | 自主决策的智能体 |
| 执行模式 | 对话 or 工作流 | 组件流编排 | 三路径：直接执行/规划执行/工作流执行 |
| 集成方式 | 平台内闭环 | MCP 发布 | API/SDK 对外输出 |
| 部署复杂度 | 重 | 轻 | 目标：轻 |

## A.3 对账中被合理拒绝的项

- **"MES 领域原生"** — 市场调研建议内置工单/设备/工艺路线模型、OPC-UA/MQTT 原生支持。用户明确拒绝，平台保持通用，通过 Skill/MCP 扩展适配领域。
- **"从内部走向外部"的长期愿景** — 简报 v2 提出"延伸为通用 Agent 编排平台，服务更多组织"。用户确认当前定位为内部产品，Non-Goals 中明确不做开放平台。
- **"能力运行时动态增减"** — 简报 v2 中"按需配置，动态组合"的表述较模糊。PRD 采用"创建时配置、运行时自主选择使用哪些能力"的模型，与当前技术实现更匹配。如果未来需要运行时增减能力，可作为后续迭代项。

## A.4 Workflow 与 Task 基础设施设计（来自 workflow-task.md）

以下内容来自 workflow-task.md 的详细设计，属于基础设施和架构层，由架构阶段负责具体实现方案。PRD 仅定义上层功能和约束。

### A.4.1 Scheduled Workflow

Scheduled Workflow 是 MVP 后续迭代的高级形态，与 Interactive Workflow 并列：

| 维度 | Interactive | Scheduled |
|------|-------------|-----------|
| 触发方式 | 用户对话 / 事件 | Cron 定时 |
| 人工节点 | ✅ 必须有 | ❌ 无 |
| 输入来源 | 外部传入（`input_schema`） | 自拉取（`prefetch`） |
| 失败处理 | 挂起等待干预 | 自动重试 + 告警 |

Scheduled Workflow 增强场景：
- 重试仍失败 → 执行 `escalation` 定义的降级动作（使用默认值/调用备用服务/发送告警到指定渠道）
- 失败会导致停线 → 自动插入"待人工确认"暂停点
- 需审批后才执行 → 模板声明 `requires_approval: true`，Task 创建后进入 `waiting_human`
- Prefetch 数据源在 `prefetch` 段声明数据源类型（HTTP/DB/File）、连接参数和轮询频率

### A.4.2 Timer 节点和 Event 节点

**Timer 节点：**
- `delay_ms` 延时触发 / `cron_expr` 定时触发
- `pause_on_suspend` 控制 Task 进入 paused/waiting_human 时是否暂停倒计时
- Timer 触发时若 Task 处于 `waiting_human` 状态：优先执行 timer 动作（如发送超时通知）

**Event 节点：**
- 必须配置 `timeout_ms`，**不允许无超时的 event 节点**
- `timeout_action` 定义超时行为：`fail`（Task 失败）/ `skip`（跳过继续）/ `default_branch`（走默认分支）
- `event_key` 声明监听的外部事件标识

### A.4.3 事件驱动架构

| 维度 | 方案 |
|------|------|
| 事件类型 | `task.node.completed` / `task.human.waiting` / `task.status.changed` / `task.completed` / `task.failed` |
| 同步路径 | Workflow Engine → 事件总线 → Task 状态机更新 → 持久化 → 推送 Session |
| 投递语义 | at-least-once（消费端需幂等去重） |
| 重试策略 | 指数退避（初始 100ms，最大 30s，最多 5 次），超出进死信队列 |
| 死信队列 | 死信事件人工/自动巡检后手动重放或丢弃 |
| 背压处理 | 事件总线缓冲区水位线告警，超阈值降级为仅持久化不同步 |
| 消费端幂等 | 事件 ID 全局唯一，消费端按事件 ID 去重 |

### A.4.4 高可用（HA）

| 组件 | HA 方案 |
|------|---------|
| Workflow Engine | 多副本部署，无状态设计，共享 Task 状态存储；单副本宕机不影响其他副本 |
| Task 调度器 | 主从模式，主节点故障自动选举 |
| 事件总线 | 基于消息队列（如 Redis Stream / RabbitMQ）的持久化队列 |
| 能力层（Tool/MCP） | 单 Tool 超时/熔断不影响其他 Tool |
| 降级策略 | Workflow Engine 整体不可用时：Direct 模式仍可用，新建 Task 排队等待 |

### A.4.5 并发与资源管理

| 维度 | 方案 |
|------|------|
| Engine 并发上限 | 单实例最大 50 并发 Task 执行（可配置），超出排队 |
| 同 Workflow 并发 | 同一模板允许最多 N 个并行 Task（N 在模板定义中声明） |
| 限流 | 全局速率限制 + 用户级配额（单用户 ≤ 5 并发 Task） |
| 熔断 | Tool 级熔断：连续失败超阈值后自动熔断，恢复后渐进重试（逐步放开并发而非立即全量恢复） |
| 乐观锁 | Task 干预操作使用 version 字段，并发冲突时提示重新获取 |

### A.4.6 审计与可观测性

| 维度 | 方案 |
|------|------|
| 审计日志 | 全量记录：Task 创建、节点执行、干预操作、状态变更，附加操作人/时间 |
| 调用链追踪 | 支持分布式 tracing（OpenTelemetry），跨 subflow 链路串联 |
| 日志保留 | 运行时日志 30 天，审计日志 90 天，可配置 |
| 监控告警 | Task 失败率、pending 队列积压、事件总线背压、Engine CPU/内存 — 统一上报 Prometheus |

### A.4.7 Workflow 模板管理机制

| 维度 | 方案 |
|------|------|
| 存储 | 数据库（MongoDB）+ 文件系统双写，DB 负责索引/查询，文件系统负责版本 diff |
| 版本化 | semver 语义版本号，每次编辑生成新版本，旧版本只读不可编辑 |
| 兼容性 | 模板版本与 Task 运行时版本解耦：已创建的 Task 永远使用其创建时的模板快照 |
| 装配执行 | 规则引擎执行装配（变量替换 + 条件节点渲染），LLM 仅参与参数推荐，不参与结构变更 |
| 迁移 | 提供 `migrate_task` 接口将运行中 Task 升级到新版本模板（需节点级兼容性检查） |
