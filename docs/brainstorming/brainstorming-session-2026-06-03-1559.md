## Architecture Design Decisions

### A. Agent 模型

Agent 是**无状态角色配置**，持久的是对话历史而非 Agent 本身。

```
Agent = 角色配置：
├── 提示词系统（模板 + 变量）
├── 工具选择（从工具池中自由组合）
├── 知识库（RAG / 文档集）
└── 运行时配置（模型、温度、Token 限制等）
```

- **Agent 之间不直接通信**，只通过 Workflow 协作
- **Agent 与 Workflow 互相嵌套**：Agent 可调用 Workflow 作为工具，Workflow 节点可以是 Agent
- **提示词变量来源统一**：Workflow 入参 / 上游节点输出 / 知识库检索 / 系统上下文 / 历史对话

### B. Workflow 编排模型

```
Workflow 编排：
├── B1. 执行语义：底层统一为有环状态图
├── B2. 前端编辑模式：
│   ├── 简单模式（DAG）— 非技术用户
│   └── 高级模式（状态图）— 开发者
├── B3. 节点类型：核心 + 插件扩展模型
│   ├── 核心：开始/结束、Agent、条件分支、并行执行
│   └── 插件：代码执行、HTTP、循环、人工审批、子工作流...
├── B4. 条件边：三层统一表达式引擎
│   ├── 节点输出判断
│   ├── 数据条件（变量表达式）
│   └── 事件触发（外部事件）
└── B5. 版本策略：修改 = 创建新 Workflow
```

### C. 用户交互层

```
用户交互：
├── C1. 入口：仅 Web UI
├── C2. 统一界面，简单模式 / 高级模式可切换
└── C3. 几十人规模，非技术用户会实际使用简单模式
```

### D. 数据与状态管理

```
状态管理：
├── D1. 数据流转：输入/输出端口模式
├── D2. 类型系统：混合型（参考 Dify，内部 Schema + 变量引用 + 自动推断）
├── D3. 持久化：
│   ├── 完整执行日志
│   └── 断点续跑（从失败节点恢复）
└── D4. 对话 ↔ Workflow 关系：
    ├── 用户对话 → Agent 可触发已配置的 Workflow
    ├── Workflow 节点 → 可触发 Agent 对话
    └── 双向触发，各自生命周期独立
```

### E. 执行引擎

```
执行引擎：
├── E1. 部署：单机
├── E2. 执行模型：异步任务队列
├── E3. 结果推送：WebSocket 实时通知
├── E4. 并发控制：任务队列 + 限流
└── E5. 循环保护：
    ├── Workflow 级最大递归深度
    └── 全局调用链深度限制
```

### F. 工具/能力层

```
工具/能力层：
├── F1. 工具与 Agent 解耦 — 工具独立存在，Agent 自由选择组合
├── F2. 统一接口 — Agent 不感知工具类型差异
├── F3. 工具来源：
│   ├── Skill（md 格式，Git 拉取 / 目录上传 / 前端创建）
│   ├── MCP（外部协议连接，配置即用）
│   └── 子工作流（系统内 Workflow 自动注册为工具，后续迭代）
└── F4. 权限控制：暂不需要，后续按需加
```

## Idea Organization and Prioritization

### 主题 1：核心架构

系统围绕 **Agent（执行单元）+ Workflow（编排逻辑）+ Tool（能力插件）** 三大核心实体构建，三者松耦合、可互相嵌套。

**关键设计亮点：**
- 工具与 Agent 解耦 → 灵活组合
- 核心 + 插件节点模型 → 强扩展性
- 底层统一状态图 + 前端双模式 → 兼顾易用性与表达力

### 主题 2：数据与交互

**关键设计亮点：**
- 输入/输出端口 + 混合类型 → 清晰的数据流转
- 对话 ↔ Workflow 双向触发 → 覆盖混合场景
- 三层统一条件边引擎 → 灵活的路由能力

### 主题 3：可靠性与安全

**关键设计亮点：**
- 异步 + WebSocket → 不阻塞，实时反馈
- 双层循环保护（Workflow 级 + 全局级）→ 防止雪崩
- 执行日志 + 断点续跑 → 故障可追溯、可恢复

### 突破性概念

- **双向触发模型**：对话和工作流不是简单的上下级关系，而是平等互触，这在同类产品中不多见
- **Skill 的归一化**：不管从 Git、目录还是前端创建，最终都是同一个 md 格式，极简

## MVP 范围定义

### MVP 包含

| 模块 | 范围 |
|---|---|
| **Agent** | 提示词 + 工具选择 + 模型配置 + 知识库 |
| **Workflow** | DAG 简单模式 + 核心节点（开始/结束/Agent/条件分支/并行） |
| **工具** | MCP + Skill（Git/上传/前端创建） |
| **对话** | Agent 对话 + 触发 Workflow |
| **编辑器** | 基础拖拽连线 + 条件边配置 |
| **持久化** | 执行日志 |
| **条件边** | 节点输出判断 + 数据条件 |

### 后续迭代

| 模块 | 内容 |
|---|---|
| **Workflow** | 高级模式（有环状态图） |
| **节点插件** | 代码执行、HTTP、循环、人工审批 |
| **工具** | 子工作流（Workflow 自动注册为工具） |
| **持久化** | 断点续跑 |
| **复用** | Workflow 模板导出/导入 |
| **转人工** | 对话中一键转人工（复用人工审批节点机制） |

## Risk Analysis (Six Thinking Hats - Black Hat)

| 风险 | 严重度 | 对策 |
|---|---|---|
| 跨 Workflow 调用链深度爆炸 | 高 | 全局 + Workflow 级双重深度限制 |
| 输入/输出端口类型系统实现复杂 | 高 | 接受投入，参考 Dify 实现 |
| 执行日志数据量增长 | 中 | 后续加清理策略 |
| MVP 功能量仍然较大 | 中 | 严格控制插件节点数量，核心节点优先 |
