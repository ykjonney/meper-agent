## 目标
用「轮询调度器」替代当前的「Celery eta 自链」，从根上消除两个 bug：
1. 占位 task 创建竞态（check-then-act，无锁）
2. eta 消息被 Redis 反复重投递（visibility_timeout 导致月度任务必然出问题）

核心设计：轮询扫 `triggers` 表 → 原子 claim（乐观锁推进 `next_trigger_at`）→ 创建占位 task → 派发**立即执行**的 Celery task（无 eta）。配合部分唯一索引做 DB 兜底。

### 第 0 步：清理脏数据（独立、安全、先做）
删除 `trig_01KX0Y8G551SNP1FVDK5AAX67S` 下 5 个重复 pending task 中的 4 个，保留 1 个。用 mongosh 原子操作。

### 第 1 步：加部分唯一索引（DB 兜底）
文件：`backend/app/services/trigger_repo.py` 的 `ensure_indexes()`
新增 tasks 集合的部分唯一索引：
```js
db.tasks.createIndex(
  { trigger_id: 1 },
  { unique: true, name: "uniq_trigger_pending_placeholder",
    partialFilterExpression: { source: "trigger", status: "pending" } }
)
```
语义：pending 时受唯一约束（同 trigger 只能 1 个）；running/completed 自动移出索引（不受约束）；done task 想存多少存多少。

### 第 2 步：`_create_placeholder_task` 改为捕获 DuplicateKeyError（消除竞态窗口）
文件：`backend/app/services/trigger_scheduler_service.py`
保留 `create_task` 的副作用（audit/event），但在 `insert_one` 外层 `try/except pymongo.errors.DuplicateKeyError`：
- 冲突 → 说明别的进程已建好占位 task，改为复用现有（update scheduled_at）后 return。
- 由唯一索引在 DB 层保证任意时刻只有一个 pending。

### 第 3 步：把 `TriggerSchedulerService` 改造成轮询调度器（核心）
文件：`backend/app/services/trigger_scheduler_service.py`
- **删除** `schedule_next()` 里的 `apply_async(eta=...)` 派发逻辑和 self-chain 调用入口。
- **新增** `_poll_loop()`（仿 `TaskSchedulerService._poll_loop`），间隔由新配置 `TRIGGER_SCHEDULER_POLL_INTERVAL`（默认 10s）控制。
- **新增** `_process_due_triggers()`：原子 claim 流程——
  ```python
  # 1. find due triggers: enabled + next_trigger_at <= now
  # 2. 对每个: 用 croniter 计算下一次 next_at
  # 3. 原子 claim: find_one_and_update({_id, next_trigger_at: 旧值}, {$set:{next_trigger_at: 新值, last_triggered_at: now}})
  #    → 返回 None 说明被别的进程抢了，skip
  # 4. claim 成功 → 创建/复用 placeholder task → 派发立即执行 Celery task（无 eta）
  ```
- **改造** `start()`：启动轮询循环 + 为"enabled 但 next_trigger_at 为空"的 trigger 计算初始 next_trigger_at（仅写 DB，不派发消息）。
- 新增辅助方法 `_compute_next(trigger, now)` 抽取 croniter/once 计算（原 schedule_next 里的逻辑）。
- 新增辅助方法 `_fire_trigger(trigger)`：创建 placeholder + 派发 execute。

### 第 4 步：简化 `execute_scheduled_workflow`（去掉 eta/self-chain 依赖）
文件：`backend/app/workers/tasks/scheduled_workflow.py`
- **删除** schedule_version 校验（轮询方案无 eta 消息需要防过期）。
- **删除** self-chain（`schedule_next(exclude_task_id=...)`）——下一次触发由轮询自动处理。
- 保留：加载 trigger/workflow → 渲染 input → 找 placeholder task → `WorkflowEngine.run_and_persist`。
- 改 Celery task 签名：不再需要 `expected_version` 参数，改为只传 `trigger_id`。

### 第 5 步：简化 API 端点（不再派发 eta）
文件：`backend/app/api/v1/triggers.py`
- `create_trigger`：若 enabled，计算初始 `next_trigger_at` 写入 DB（不调 schedule_next，不派发消息）。
- `update_trigger`：重新计算 `next_trigger_at` 写入（可保留 schedule_version bump 用于调试，但不再用于防 eta 过期）。
- `toggle_trigger`：enable 时计算 next_trigger_at；disable 时保留现状（轮询靠 enabled 过滤）。
- 三个端点都不再调用 `scheduler.schedule_next(...)`。

### 第 6 步：新增配置项
文件：`backend/app/core/config.py`
```python
TRIGGER_SCHEDULER_POLL_INTERVAL: int = 10  # seconds; 0 = disabled
```

### 第 7 步：清理不再需要的配置
文件：`backend/app/workers/celery_app.py`
- visibility_timeout 的 7 天 hack 注释说明它原本是为 eta。改为保留默认或加注释说明"trigger 已不依赖 eta，此值仅供其它 eta 场景"。**不强制改动**，避免误伤其它用途。

## 兼容性与风险
- `schedule_version`、`celery_task_id` 字段保留在 model 中（不删，避免破坏已有数据），只是部分不再写入。
- 现有 done/failed/cancelled task 数据不受影响。
- 精度从"eta 精确到秒"变为"轮询间隔 10s"，对定时任务可接受。
- main.py lifespan 无需改动（仍调 `trigger_scheduler.start()`，只是 start 内部行为变了）。
